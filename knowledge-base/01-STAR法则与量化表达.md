# STAR法则与量化表达

## 一、STAR法则核心

STAR法则是简历项目描述的黄金框架，每一个项目经历都应按照以下结构组织：

| 组件 | 全称 | 含义 | 简历中的角色 |
|------|------|------|-------------|
| **S** | Situation | 背景/场景 | 项目的业务背景、规模、痛点（1-2句话） |
| **T** | Task | 任务/目标 | 你需要解决的问题或达成的目标 |
| **A** | Action | 行动/措施 | **你**具体做了什么（技术方案+关键决策） |
| **R** | Result | 结果/影响 | 量化的成果（数据、指标、业务价值） |

### 正确示例

> **S/T**: 负责公司核心订单系统从单体架构向微服务架构的迁移，日均订单量 50万+，原有系统在促销期间响应时间超过 3秒。
>
> **A**: 基于 Spring Cloud + Dubbo 设计服务拆分方案，将订单服务拆分为 12 个微服务；采用 Seata 实现分布式事务一致性；通过 RocketMQ 实现订单异步削峰；设计灰度发布策略确保平滑迁移。
>
> **R**: 系统峰值 QPS 从 2000 提升至 15000，P99 响应时间从 3.2s 降至 180ms，订单处理成功率从 99.5% 提升至 99.99%。

### 错误示例

> 负责订单系统微服务化改造，使用 Spring Cloud 和 Dubbo 进行服务拆分，提高了系统性能。

**问题**: 没有 S/T 背景、A 没有具体方案、R 没有量化数据。

## 二、动词选择清单

简历中每条描述应以**强动词**开头，避免"负责""参与"等弱动词。

### 推荐动词（按技术场景分类）

**架构设计类**
- 设计、重构、主导、规划、架构、定义

**开发实现类**
- 实现、开发、构建、搭建、编写、交付

**优化改进类**
- 优化、提升、改进、加速、精简、重构

**运维部署类**
- 部署、运维、监控、配置、自动化、迁移

**分析研究类**
- 分析、评估、调研、验证、排查、定位

**管理协作类**
- 推进、协调、带领、指导、评审、制定

### 动词强度排序

| 强度 | 动词 |
|------|------|
| ★★★★★ | 主导、架构、从0到1搭建、重新设计 |
| ★★★★ | 设计、重构、优化、推进、交付 |
| ★★★ | 开发、实现、搭建、维护 |
| ★★ | 参与、协助、配合 |
| ★ | 负责（除非后面跟着强结果） |

## 三、量化表达的六种模式

### 1. 性能指标
- QPS/TPS: "QPS 从 2000 提升至 15000"
- 响应时间: "P99 延迟从 3.2s 降至 180ms"
- 吞吐量: "日处理消息量从 100万提升至 500万"

### 2. 业务指标
- 用户量: "支撑 DAU 500万+ 用户"
- 交易额: "双11 当日处理交易额 12亿元"
- 转化率: "支付转化率从 65% 提升至 78%"

### 3. 效率指标
- 开发效率: "接口交付周期从 5天缩短至 1天"
- 部署频率: "部署频率从每周1次提升至每日3次"
- 故障恢复: "MTTR 从 45分钟缩短至 5分钟"

### 4. 质量指标
- 可用性: "系统可用性从 99.9% 提升至 99.99%"
- 错误率: "接口错误率从 2% 降至 0.1%"
- 覆盖率: "单元测试覆盖率从 30% 提升至 85%"

### 5. 规模指标
- 数据量: "管理 PB 级数据存储"
- 服务数: "管理 50+ 微服务"
- 团队规模: "带领 8人后端团队"

### 6. 成本指标
- 服务器成本: "服务器成本降低 40%"
- 资源利用率: "CPU 利用率从 20% 提升至 65%"

## 四、技术栈的分层表达

不要平铺罗列技术栈，按层次组织更能体现架构视野：

```
架构层:  Spring Cloud / Dubbo / K8s / Service Mesh
服务层:  Spring Boot / Go-Micro / gRPC
数据层:  MySQL / Redis / Elasticsearch / MongoDB
中间件:  RocketMQ / Kafka / ZooKeeper
基础设施: Docker / Jenkins / Prometheus / Grafana
```

### 技术栈描述模板

> 技术栈: Java 17 + Spring Cloud Alibaba + MySQL 8.0 + Redis Cluster + RocketMQ + Elasticsearch 8.x + K8s

**注意**: 只列实际深度使用的，不堆砌名词。

## 五、项目经历的结构模板

每个项目经历建议包含以下要素：

```markdown
### [项目名称]
**时间**: 2023.06 - 2024.03
**角色**: 后端负责人 / 核心开发
**技术栈**: [分层列出]

**项目背景**:
[S/T: 业务背景 + 痛点 + 目标]

**核心工作**:
- [Action 1 + Result 1]
- [Action 2 + Result 2]
- [Action 3 + Result 3]

**技术亮点** (可选):
[最值得面试深入聊的技术难点]
```

## 六、篇幅建议

| 经验年限 | 项目数量 | 每项目篇幅 | 总简历页数 |
|----------|----------|-----------|-----------|
| 1-3年 | 2-3个 | 5-8行 | 1页 |
| 3-5年 | 3-4个 | 6-10行 | 1-2页 |
| 5-8年 | 3-4个 | 8-12行 | 2页 |
| 8年+ | 3-4个 | 8-15行 | 2-3页 |

**原则**: 质量优于数量。2个有深度的项目 > 5个走马观花的项目。

## 七、自我评价写法

### 推荐结构（3句话）

1. **定位句**: X年经验 + 技术领域 + 核心能力
2. **亮点句**: 最突出的1-2个技术深度或业务成果
3. **软实力句**: 团队协作/开源贡献/技术影响力

### 示例

> 5年后端开发经验，精通 Java 微服务架构与高并发系统设计。主导过日均千万级订单系统的架构演进，具备从0到1搭建中台能力。GitHub 开源项目 2k+ stars，活跃于技术社区。

### 避免

- "吃苦耐劳、热爱学习" → 空话，每个候选人都这么写
- "精通 XX" → 除非真的精通，否则用"深入理解"或"熟练掌握"
- 过长的自我评价 → HR 只看 5 秒
