**后端**：Python + FastAPI  
**RAG 框架**：LangChain  
**第一版多模态输入**：文本 + 图片 + PDF  
**问答模型**：支持图像理解的模型  
**检索层**：先做“文本主导检索”，图像理解用于抽取和补强  
**向量库**：FAISS  
**工作流**：先不用 LangGraph，第二版再加

为什么这么配：

- FastAPI 原生适合文件上传，官方也明确说明文件通过 `File` / `UploadFile` 处理，适合 API 服务。
- LangChain 目前依然把简单 RAG 链路作为起步方案，复杂 agentic RAG 属于后续升级路线。
- 你现在最缺的是把**多模态输入接进 RAG**，而不是先上复杂图编排。

#### 你应该怎么理解“多模态版 RAG”

多模态版你要这样理解：

文本 / 图片 / PDF  
-> 解析与理解  
-> 统一变成可检索知识单元  
-> embedding / 建索引  
-> 用户问题  
-> 检索相关知识单元  
-> 把文本上下文 + 必要视觉内容交给模型  
-> 回答

重点不在“所有东西都直接向量化”，而在于：

> **先把图片/PDF 里的信息变成系统能稳定利用的知识单元。**

对于你现在的阶段，最稳的做法不是直接上复杂 multimodal embedding，而是：

### 推荐做法

1. **图片/PDF 先走视觉理解**
    - 提取摘要
    - 提取文字
    - 提取表格/字段
    - 生成结构化描述
2. **把这些结果转成文本知识单元**
    - 再做 chunking
    - 再做 embedding
    - 再进 FAISS
3. **需要时把原图/页面再送给视觉模型二次确认**
    - 尤其是表格、图表、扫描件、布局敏感内容

OpenAI 最近的文档理解建议也在强调，很多过去要 OCR + 布局分析 + 自定义管线拼起来的任务，现在常常可以先通过单次视觉理解完成，再根据任务做后续处理。

# 六、每种文件怎么处理

## 1. 纯文本

最简单：

- 直接加载
- chunking
- embedding
- 入库

## 2. 图片

图片不要直接粗暴做“纯 OCR 然后入库”。  
第一版更推荐：

### 图像理解输出两层结果

- **层 1：可检索摘要**
    - 这张图是什么
    - 有哪些关键元素
    - 是否包含文字、表格、截图、报表
- **层 2：结构化字段**
    - 标题
    - 关键词
    - 识别出的文字
    - 表格/图表摘要
    - 文件来源

然后把这些结果转成文本片段入库。

为什么这样更好：

- 纯 OCR 对截图、表格、流程图、界面类内容不够
- 视觉理解能补足“图像语义”
- 而文本化后，检索与 RAG 主链仍然保持简单稳定

OpenAI 最近的视觉与文档理解资料明确指出，很多复杂文档与图像内容可以直接在模型中完成理解与推理，而不只是做字符识别。

## 3. PDF

PDF 分两类：

### 可提取文本的 PDF

- 优先抽文本
- 保留页码
- 对页内容做 chunking

### 扫描件 / 图文混排 PDF

- 按页渲染
- 每页走视觉理解
- 每页生成“文本化页面摘要”
- 必要时保留页面图片引用

这一类特别适合你的多模态版项目，因为“页级理解 + 页码引用”会很像真实企业文档问答。

# 七、RAG 主链怎么写

对你当前阶段，主链仍然建议用最简单稳定的方式：

- `RecursiveCharacterTextSplitter`
- `OpenAIEmbeddings`
- `FAISS`
- `retriever`
- `create_stuff_documents_chain`
- `create_retrieval_chain`

LangChain 当前最小 RAG 仍然是这个思路。

但多模态版要多一个“前处理层”：
```
文件上传
-> 文件分类
-> 文本抽取 / 图像理解 / PDF页理解
-> 统一转成 Document
-> chunking
-> embedding
-> FAISS
-> retriever
-> augment
-> LLM 回答
```
所以你真正新增的不是 RAG 主链，而是：

> **多模态内容 -> 文本化知识单元** 这层。

# 八、接口应该怎么设计

## 1. 上传知识文件

`POST /ingest/file`

表单：

- `file`
- `kb_name`
- `file_type`（可选，不传则自动判断）

FastAPI 官方说明，文件与表单字段要用 `File` 和 `Form`，因为这类请求是 `multipart/form-data`。

返回：
```JSON
{
  "message": "文件导入成功",
  "file_type": "pdf",
  "chunks": 18,
  "pages": 6
}
```

## 2. 问答接口

`POST /chat`

请求：
```JSON
{
  "question": "这份文档里关于报销上限怎么规定？"
}
```
返回：
```JSON
{
  "answer": "根据文档内容，报销上限为……",
  "sources": [
    {
      "source": "expense_policy.pdf",
      "page": 3,
      "content": "……"
    }
  ]
}
```
## 3. 图片单问答

`POST /chat/image`

用途：

- 用户上传一张图直接提问
- 不一定入知识库

请求：

- `file`
- `question`

这个接口适合做即时视觉问答，不一定走完整 RAG。

# 十、我不建议你现在做什么

### 1. 不建议先做语音助手

虽然 OpenAI 现在官方支持音频与语音 agent，并且已有专门的音频/语音文档，但你当前最容易成功的不是语音，而是文档/图片问答。

### 2. 不建议先做多模态 embedding 全量检索

你现在更应该先做：

- 图像/PDF 理解
- 文本化后检索

因为这条工程路径更稳，更容易 debug，也更符合你当前的知识积累。

### 3. 不建议先做 LangGraph

你还没把多模态输入链跑顺，先别上复杂编排。
